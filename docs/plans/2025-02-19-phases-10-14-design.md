# Phases 10-14: Design Document

**Date:** 2025-02-19
**Status:** Approved
**Prerequisite:** Phase 9 (CLI) complete

---

## Decision Summary

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Persistence integration point | API layer only | Core engine stays pure. CLI works without DB. |
| Database | PostgreSQL only | No SQLite. Dev uses `docker compose up db`. |
| Table scope | 5 core tables | Training data tables deferred until UI validation exists. |
| API authentication | API key header | Simple `X-API-Key`. Sufficient for self-hosted v1. |
| Ingest format | Raw transcripts | Server parses via registry. Zero friction for integrators. |
| Report output | Markdown + JSON | PDF deferred. No LLM narrative for v1. |
| Graph visualisation | React Flow | Purpose-built for node DAGs. TypeScript first. |
| Component library | shadcn/ui | Copy-paste components. Tailwind. No runtime lock-in. |
| UI scope | Full investigation view | Session list, graph, inspector, validation, reports. |
| Deployment model | Single container | FastAPI serves React static. Multi-stage Docker build. |

## Phase 10: Database Models and Persistence

### Scope

SQLAlchemy ORM models, Alembic migrations and a persistence service layer for the 5 core tables.

### Database

PostgreSQL only. Local dev uses `docker compose up db`. Tests use testcontainers or pytest-postgresql.

### Tables

1. **sessions** - Agent workflow executions (UUID PK, external_id, agent_id, timestamps, status, JSONB metadata)
2. **decision_nodes** - Core analysis unit (UUID PK, FK to session, self-referencing parent FK, sequence_num, event_type, action, JSONB inputs/outputs/metadata, 5 boolean risk flags, is_inflection_node)
3. **recurrence_signatures** - Cross-session patterns (signature_hash, JSONB pattern, occurrence_count, severity)
4. **session_signatures** - Junction table linking sessions to signatures (UUID[] matched_nodes)
5. **reports** - Generated forensic reports (session FK, report_type, content_markdown, content_json)

Training data tables (inflection_validations, risk_flag_validations, signature_validations, telemetry_exports) are deferred until the UI validation workflow exists.

### Integration Pattern

API layer only. `analyze_session()` stays pure. A `PersistenceService` class handles mapping `AnalysisResult` to ORM models. The CLI path never touches the database.

### Module Structure

```
src/driftshield/
  db/
    models.py        # SQLAlchemy ORM models
    engine.py        # Engine/session factory, connection config
    persistence.py   # PersistenceService (save/load AnalysisResult)
    migrations/      # Alembic directory
      env.py
      versions/
```

### Indexes

- `parent_node_id` on decision_nodes (traversal)
- `session_id + sequence_num` on decision_nodes (ordering)
- `signature_hash` on recurrence_signatures (lookups)
- `session_id` on reports

### On-site Deployment

PostgreSQL is part of the Docker Compose stack (Phase 14). Customers run `docker compose up` and get both the app and a Postgres container with persistent volumes. Organisations with an existing managed Postgres instance point the config at their DB. Alembic migrations handle schema setup either way.

## Phase 11: API Routes (FastAPI)

### Scope

FastAPI application with REST endpoints for ingestion, session browsing, graph retrieval and report generation.

### Authentication

`X-API-Key` header checked by FastAPI middleware. Single shared key from config/environment variable. Requests without a valid key get 401.

### Routes

| Method | Route | Purpose |
|--------|-------|---------|
| `POST` | `/api/ingest` | Upload raw transcript file. Server detects format, parses, analyses, persists. Returns session ID + summary. |
| `GET` | `/api/sessions` | List sessions with pagination, filtering (status, agent_id, date range), sorting. |
| `GET` | `/api/sessions/{id}` | Get session detail including analysis result. |
| `GET` | `/api/sessions/{id}/graph` | Get lineage graph data (nodes + edges) for React Flow rendering. |
| `POST` | `/api/sessions/{id}/report` | Trigger report generation. Returns report ID. |
| `GET` | `/api/sessions/{id}/reports` | List reports for a session. |
| `GET` | `/api/reports/{id}` | Get a specific report (Markdown or JSON based on Accept header). |
| `GET` | `/api/health` | Health check (DB connectivity, app version). |

### Ingest Flow

1. Receive multipart file upload + optional `format` parameter (default: `auto`)
2. Detect format via parser registry (reuses Phase 9 parsers)
3. Parse to CanonicalEvents
4. `analyze_session()` runs the pipeline
5. `PersistenceService` saves results to DB
6. Return `201` with session ID and summary

### Pagination

Offset-based with `?page=1&per_page=20` (default 20, max 100). Response includes `total`, `page`, `per_page`, `pages`.

### Module Structure

```
src/driftshield/
  api/
    app.py           # FastAPI app factory, middleware, lifespan
    auth.py          # API key middleware
    routes/
      ingest.py      # POST /ingest
      sessions.py    # Session CRUD + graph
      reports.py     # Report generation + retrieval
      health.py      # Health check
    schemas.py       # Pydantic request/response models
    dependencies.py  # DB session, auth, shared deps
```

### Error Handling

Standard error responses with `{"detail": "message", "code": "ERROR_CODE"}`. HTTP status codes: 400 validation, 401 auth, 404 not found, 422 parse failure, 500 internal.

### No WebSocket for v1

Analysis is synchronous on ingest. Async processing is a later concern.

## Phase 12: Report Generation

### Scope

Template based forensic report generation producing Markdown and JSON outputs. Integrated with both API and CLI.

### Output Formats

- **Markdown** - Human readable forensic report. Primary output.
- **JSON** - Structured equivalent for UI rendering.
- **PDF** deferred to a later phase.

### Template Engine

Jinja2. Templates are plain Markdown files with Jinja2 placeholders, shipped as part of the Python package.

### Report Structure

1. **Behavioural Lineage Reconstruction** - Chronological decision node table with risk flags
2. **Inflection Node Identification** - Where reasoning diverged, confidence level, analysis
3. **Risk State Transition Mapping** - How risk propagated through the graph
4. **Systemic Exposure Assessment** - Isolated / Recurring / Systemic classification
5. **Recurrence Risk Analysis** - Probability, contributing factors, recommendation

### Report Types

- `full` - All 5 sections. Default.
- `summary` - Sections 1 + 2 only. Quick overview.
- `comparison` - Side by side comparison of two sessions. Deferred.

### Data Flow

```
AnalysisResult + Session data
       ↓
  ReportBuilder
       ↓
  ReportData (structured Python object)
       ↓
  ┌────────────┬──────────────┐
  Jinja2       JSON serialiser
  ↓            ↓
  Markdown     JSON
```

### No LLM Narrative for v1

Reports are purely data driven from analysis results. LLM assisted narrative is a Phase 2 feature.

### CLI Integration

`driftshield report <session-path>` generates a Markdown report to stdout or file. Uses the same `ReportBuilder` as the API. No database needed for CLI path.

### Module Structure

```
src/driftshield/
  reports/
    builder.py       # ReportBuilder - assembles ReportData from AnalysisResult
    models.py         # ReportData, ReportSection dataclasses
    markdown.py       # Jinja2 rendering to Markdown string
    json_export.py    # Structured JSON serialisation
    templates/
      full.md.j2      # Full report template
      summary.md.j2   # Summary report template
```

## Phase 13: React UI Foundation + Investigation View

### Scope

Full investigation interface. Session browser, lineage graph visualisation, node inspector, risk flag display, analyst validation controls, report generation trigger and preview.

### Tech Stack

- React 18 + TypeScript
- Vite (build tooling)
- React Flow (lineage graph)
- shadcn/ui + Radix primitives (components)
- Tailwind CSS (styling)
- TanStack Query (server state / data fetching)
- React Router (routing)

No heavy state management library. TanStack Query handles server state. Local UI state lives in React context or component state.

### Module Structure

```
driftshield/
  frontend/
    src/
      components/
        layout/
          AppShell.tsx
          Header.tsx
        sessions/
          SessionList.tsx
          SessionRow.tsx
          SessionFilters.tsx
        investigation/
          InvestigationView.tsx
          LineageGraph.tsx
          GraphNode.tsx
          GraphEdge.tsx
          NodeInspector.tsx
          RiskFlagBadge.tsx
          InflectionMarker.tsx
        validation/
          ValidationControls.tsx
          ValidationHistory.tsx
        reports/
          ReportTrigger.tsx
          ReportPreview.tsx
      pages/
        SessionListPage.tsx
        InvestigationPage.tsx
      api/
        client.ts
        sessions.ts
        reports.ts
      types/
        session.ts
        graph.ts
      App.tsx
      main.tsx
    package.json
    vite.config.ts
    tailwind.config.ts
    tsconfig.json
```

### Key Views

**Session List** (`/sessions`)
- Table: session ID, agent, status, risk flag count, inflection detected, timestamp
- Search by session ID or agent
- Filter by status, date range, risk severity
- Sort by any column
- Click row to open investigation view

**Investigation View** (`/sessions/:id`)
- Split layout: lineage graph (left/main) + node inspector (right panel)
- **Lineage graph:** React Flow DAG, top-to-bottom layout (dagre). Nodes coloured by risk state. Inflection node visually distinct. Click node to select.
- **Node inspector:** Selected node's full detail. Event type, action, inputs, outputs, metadata. Risk flags with explanations. Parent/child navigation.
- **Validation controls:** Below inspector. Confirm or reject inflection identification. Validate or dispute risk flags.
- **Report trigger:** Button in header. Select report type. Generate and preview. Download as Markdown.

### Graph Node Design

```
┌─────────────────────┐
│ ● TOOL_CALL         │  ← event type badge
│ extract_clauses     │  ← action name
│ ⚠ coverage_gap     │  ← risk flags (if any)
└─────────────────────┘
```

Inflection nodes get a distinct border/glow. Nodes with risk flags show warning indicators.

### Validation Workflow

1. Analyst views investigation
2. Clicks inflection node → inspector shows "Confirm as inflection?" / "Reject"
3. Clicks risk flag → "Validate" / "Dispute" / "Mark false positive"
4. Decisions POST to API → stored (training data tables added when needed)
5. Graph updates to reflect validated state

### Deferred

Comparison view (side by side session comparison) deferred to a follow-up phase.

## Phase 14: Docker Deployment

### Scope

Docker Compose self-hosted deployment. Single container for the app, PostgreSQL container, configuration management.

### Architecture

```
docker-compose.yml
  ├── app     (Python + React static, port 8080)
  └── db      (PostgreSQL 16, port 5432)
```

### Multi-stage Dockerfile

```
Stage 1: frontend-build (node:20-alpine)
  → npm ci && npm run build → /app/dist

Stage 2: backend (python:3.12-slim)
  → pip install from pyproject.toml
  → COPY --from=frontend-build /app/dist → /app/static
  → Alembic migrations run on startup
  → Uvicorn serves FastAPI
```

FastAPI serves everything:
- `/api/*` → API routes
- `/*` → React static files (SPA fallback to index.html)

### Configuration

Environment variables with sensible defaults.

| Variable | Default | Purpose |
|----------|---------|---------|
| `DATABASE_URL` | `postgresql://drift:drift@db:5432/driftshield` | DB connection |
| `API_KEY` | (required, no default) | Authentication key |
| `LOG_LEVEL` | `info` | Logging verbosity |
| `WORKERS` | `1` | Uvicorn worker count |
| `HOST` | `0.0.0.0` | Bind address |
| `PORT` | `8080` | Bind port |

### Database Initialisation

1. Container starts
2. Entrypoint script waits for PostgreSQL to be ready (pg_isready loop)
3. Runs `alembic upgrade head` to apply migrations
4. Starts Uvicorn

### Health Checks

- App container: `GET /api/health` (checks DB connectivity)
- DB container: `pg_isready`

### Volumes

- `postgres_data` named volume for database persistence
- Optional bind mount for transcript files if using local file ingestion

### Docker Compose Profiles

Single `docker-compose.yml` with a `dev` profile override.

- **Default (production):** App + DB, no source mounts, migrations on start.
- **Dev profile:** Source bind mount, hot reload (`--reload`), frontend dev server on separate port, local `.data/` directory for Postgres data.

### Module Structure

```
driftshield/
  Dockerfile
  docker-compose.yml
  docker/
    entrypoint.sh      # Wait for DB, run migrations, start Uvicorn
    .env.example        # Template for environment variables
```

### .dockerignore

Standard exclusions: `.venv`, `node_modules`, `.git`, `__pycache__`, test fixtures, docs.
